问题1：
现象：右边电机仅在 PWMR < 0 才转，PWMR ≥ 0 不转。
原因：正转需 PB14 高电平，但因硬件故障（短路/虚焊）PB14 被强制拉低，方向信号错误致电机不转；反转 PB14 本就为低，故正常。
解决：更换 PB14 引脚或修复硬件。

左右控制交叉，PWM/方向引脚接线交叉，按代码逻辑重接。

day3
今天绝对是糟糕的一天，我本以为是PB14虚焊导致的1号电机不反转今天收到万用表后发现PB14并没有虚焊，于是我开始寻找原因以下是我的总结。
1. 初始软件 Bug
按键逻辑错误：KeyNum==2 和 KeyNum==4 中误将 PWMR/PWML 的下限判断写成 >100，且变量名混淆，导致 PWML、PWMR 无法正确变为负数。
Motor_SetPWM 缺少绝对值限制：比较值可能超出 TIM_Period 范围 (0~99)，存在潜在异常。
2. 核心现象
PWML > 0 时左电机正转正常，PWML < 0 时左电机不反转（右电机正反转均正常）。
3. 排查过程与关键发现
单独 MCU 测试：PB14 可正常输出 3.3V 高电平 → MCU 引脚本身完好。
插入电路板后：PB14 电平被强制拉低至 0V，无法输出高电平。
电路板侧检查：PB14 到电机驱动 BIN1 的线路用蜂鸣档测通断正常 → 没有虚焊/断线。
驱动板 BIN1 对地电阻测量：187kΩ，正常，无内部强下拉或短路。
最终硬件结论：原驱动板存在 隐性 PCB 断裂/过孔不良，导致 MCU 到驱动端的信号时通时断（驱动端浮空时被内部下拉到 0V），因此恰好“正转需要低电平，电机转；反转需要高电平，电机不转”。
4. 附加故障
排查过程中不慎 电源接反，驱动板烧毁，所有测试终止。
5. 最终结论与解决方案
根本原因：驱动板 PCB 连接不可靠（疑似过孔或焊盘内部断裂），而非软件配置或引脚占用问题。
后续行动：
更换新的电机驱动板。
新板焊接后，用万用表电阻档验证 MCU 至驱动板输入端电阻 <1Ω，确保连通。
电源端增加防反接措施（如串联肖特基二极管或自恢复保险）。
恢复原先的代码逻辑（方向引脚仍用 PB14/PB15），软件仅需修正初始按键 Bug 和绝对值限制即可正常工作。

day4
今天打算重焊一张板 焊了最小系统板后发现GPIOB14仍然无法输出高电平于是有了以下的步骤
1. 现象描述：诡异的“消失的高电平”
在组装平衡车底板时，遇到一个经典的硬件“玄学”问题：
软件配置： 将 STM32 的 GPIOB14（PB14）配置为推挽输出高电平（3.3V）
拔下测试： 单片机悬空测试 PB14，万用表显示完美的 3.3V。
插上底板： 将单片机插到底板母座上，PB14 电压瞬间被拉低至接近 0V，无法正常输出高电平。
 2. 排错全过程
Step 1：初步隔离与阻值排查
最初怀疑是焊接母座时导致了“连锡”短路。
在断电状态下，使用万用表“蜂鸣档”测试 PB14 与 GND，未发出蜂鸣声。
切换至“电阻档”测试，发现 PB14 与 GND 之间存在几百欧姆且不断跳动的阻值。
初步结论： 存在“漏电”性质的微弱短路，可能是助焊剂残留或微小的寄生电容/电阻瞬间吸干了 MCU 极弱的驱动电流。
Step 2：PCB 走线审查（发现首个设计缺陷）
由于连续两块同样的 PCB 都出现此问题，排除了偶然的焊接失误，转向审查 PCB 走线图（Layout）。
发现致命雷区： PB14 的顶层走线，强行从对面排针的 GND 和 3.3V 焊盘的极小缝隙中“穿缝”而过。
推论： 间距过窄，在焊接排针母座时，高温烫化了 PB14 走线上的阻焊油，导致焊锡或助焊剂渗透，引发 PB14 与紧邻的 GND 产生漏电短路。
Step 3：割线与飞线替换（陷入僵局）
为了绕过 PB14 的物理缺陷，决定实施“割线飞线”大法：
切断底板上 PB14 的走线。
修改代码，将输出引脚改为底板上没有走线、完全悬空的 PC15（C15 虽然驱动能力限制在 3mA，但作为逻辑信号输入绰绰有余）。
意外发生： 飞线将 C15 连接到目标引脚（TB6612 的 BIN1）后，C15 的高电平同样被拉低了！
Step 4：排查终点与最终破案
C15 的失败完美洗清了单片机侧的嫌疑，问题绝对出在终点 BIN1 或连接模块上。
终极测试： 拔下所有外设，单独测量此前因为电池过压/接反被烧毁过的 TB6612 模块。
真相大白： 被烧毁的 TB6612 芯片内部逻辑输入端（BIN1）已经被彻底击穿并对地（GND）短路。
补充暴击（厂家制造缺陷）： 拿出一块完全未焊接的全新裸板测试，发现出厂状态下 PB14 和 BIN1 就已经对 GND 短路（蜂鸣档长响）。原因是厂家蚀刻不干净，极其狭窄的线距导致出厂“连铜”。
 3. 根本原因总结 (Root Cause)
本次 Bug 是一个极其罕见的“双重硬件故障叠加”：
PCB 制造与设计缺陷： PB14 走线穿缝间距过小，导致出厂残铜短路以及后期焊接高温漏电。
外围器件内部击穿： TB6612 电机驱动模块因早期供电异常烧毁，导致输入端 BIN1 内部对地死短路，形成了“电平黑洞”。
 4. 最终解决方案
物理隔离： 在新板子上正常焊接母座，但将新买的 TB6612 模块的 BIN1 排针用钳子向外掰弯 90 度，使其悬空，不插入短路的底板孔位。
飞线绕过： 修改软件代码，使用无走线干扰的 PC15 或 PB5 作为新的控制引脚。
空中连接： 飞线直接连接 MCU 底部的新引脚与悬在空中的 TB6612 BIN1 针脚，完美避开底板上的所有短路雷区。

5.9-5.12开始调试速度环期间遇到了很多的问题以下是我遇到后思考解决不了询问ai的：
坑 1：“补丁式”加负号的混乱深渊初始现象： 无论怎么改速度环的参数，效果都不对。发现反了，就在代码里随便找个地方加个 -1（比如电机输出加负号、编码器取反、角度计算加负号）。根本原因： 系统“前进”的物理定义在硬件和软件中根本不一致！这就导致了只要在一个地方打补丁改符号，其他地方就要反向补偿，最后代码里的极性逻辑变成了一团乱麻。破局之道： 停止在 PID 里瞎凑符号！先写个开环测试程序，彻底定标物理方向：强迫 正PWM = 车轮向前转 = 编码器读数为正 = 倾角为正。 硬件基准统一后，直立环Kp 顺理成章定为负值（如 -3.12），速度环目标定为 Target = SpeedPID.Out（或者加负号，取决于你的极性定义，但全车必须只有一套标准）。
坑 2：速度环最大的直觉陷阱（正负反馈之争）现象： 引入速度环后，小车不仅不减速，往前推它反而加速往前冲。核心顿悟： 调普通小车，超速了给负反馈（踩刹车）。但平衡车想减速，必须先加速！ 发现往前跑太快，电机必须输出正向的加速 PWM，让轮子跑到重心的前面去，强迫车身向后仰，利用重力来刹车。速度环在宏观表现上是一个正反馈过程。
坑 3：手持稳如老狗，下地秒变废铁（直立内环没调好）现象： 手持悬空时速度环看着挺正常，一放地上车就倾倒，或者往一个方向狂奔。根本原因： 你的纯直立环（内环）本身就站不住！如果机械零点偏了，或者内环刚度不够，一旦下地加上摩擦力就原形毕露了。解决：细调机械零点（AngleAcc += 1.80），让纯直立环下地时尽量不单向加速。给直立环也加入微小的 I项（并做好积分限幅），让它能自己克服地面的微小扰动站稳。
坑 4：限幅的博弈（太大直接摔死，太小无力回天）现象： 速度环限幅给得太小，小车前倾角度稍大，速度环目标倾角就被钳位，根本产生不了足够的后仰指令去拉回车身；但如果给太大，小车为了刹车会瞬间倾斜过大直接超载倒地。解决： 速度环的 OutMax/Min 必须卡在物理极限的甜点区，确保既有足够力量拉回，又不会做出“自杀式”杂技动作。
坑 5：机械死区与被 $Ki$ 支配的恐惧（积分饱和）现象： 直立环能自己站稳了，但加入速度环的 $Ki$ 后，小车会缓慢向一个方向倾斜直至倒下。或者出现“大摆钟式”的跨零点疯狂前后摇摆。根本原因：机械死区： 轮子和联轴器存在约 2° 的物理虚位（死区）。在这 2° 内编码器没有速度反馈，但 $Ki$ 却在暗中疯狂累加误差。一旦突破死区，系统就会产生过激反应。积分饱和（Windup）： 只要速度没归零，积分项就一直把目标角度推向极限，最终导致车体猛烈倾斜。救命稻草：加入积分分离逻辑： 当速度误差过大（比如被猛推时），直接清零积分项，只有当误差较小、接近稳态时才允许 $Ki$ 介入。绝对的积分限幅： 速度环的积分累加值必须被死死按在极小范围内。
🗺️ 附录：唯一正确的参数调节路径以前总喜欢直接给大参数找感觉，或者 P/I/D 一起上，这在平衡车上是死路一条。请严格遵守以下步骤：统一物理基准： 写开环代码，统一电机、编码器、陀螺仪的正方向。死磕直立内环： Kp、Kd、微小 I 项，外加精准的零偏补偿。必须做到无速度环时，小车下地能勉强自己站稳或平缓匀速溜车。引入速度 P 项： 加上合理的速度环限幅，慢慢给 SpeedKp。目标是：推它一下，能干脆抵抗并变成匀速溜车（不再出现大幅度穿过零点的荡秋千）。注入速度 I 项： 开启积分限幅和积分分离，给入微小的 SpeedKi。消灭最后的匀速溜车稳态误差，让小车“推不倒，推了自己退回来”。硬件决定下限： 如果参数怎么调都有问题，去检查联轴器虚位、重心偏移。机械不稳，PID 是托不住底的。
